# gstack 七大角色深度解析

**来源**：gstack（Y Combinator CEO Garry Tan 开源工具）
**分析日期**：2026-03-31
**适用场景**：AI 开发工作流、代码审查、安全审计、设计评审

---

## 总览

| 角色 | Emoji | 核心职责 | 输出物 |
|------|-------|---------|--------|
| 👔 CEO | 产品战略 | 方向把控、优先级排序 | 产品路线图、功能评审 |
| 🏗 工程经理 | 架构决策 | 技术选型、系统设计 | 架构文档、代码规范 |
| 🎨 设计师 | 设计质量 | 识别 AI 设计缺陷 | 设计规范、设计评审 |
| 🔍 评审员 | 代码审查 | Bug 发现、安全漏洞 | PR 评审报告 |
| 🧪 QA 负责人 | 测试验证 | 浏览器自动化测试 | 测试报告 |
| 🔐 安全官 | 安全审计 | OWASP/STRIDE 威胁建模 | 安全报告 |
| 🚀 发布工程师 | 部署运维 | CI/CD 自动化、发布 | 部署流水线 |

## 1. 👔 CEO（产品战略）

### 职责定义
- 站在产品视角评估功能方案
- 质疑需求的必要性和优先级
- 防止"功能膨胀"和技术债堆积
- 确保做的东西真正有价值

### 触发场景
```
用户: 我想加一个聊天表情搜索功能
CEO: 为什么要搜？现有表情不够用？还是选图效率低？
     这个问题用手势和分类导航能不能解决？
     如果要做，核心用户是谁？日活多少？
```

### 常用技能
- `/plan-ceo-review` — CEO 视角评审功能方案
- `/office-hours` — 产品方向咨询

### 核心原则
> "这个功能删掉会对用户有什么损失？如果没有，它就不该加。"
> —— 类似于"简约至上"原则

---

## 2. 🏗 工程经理（架构决策）

### 职责定义
- 把控技术选型和架构方向
- 锁定核心设计决策，防止架构漂移
- 制定代码规范和最佳实践
- 评估技术债务和重构时机

### 触发场景
```
用户: 我想用 MongoDB 替代 MySQL，因为文档存储更灵活
EM: 你们的核心查询模式是什么？关联查询多吗？
    团队有 MongoDB 运维经验吗？
    换数据库的迁移成本是多少？
    建议先做 POC，不要全量替换
```

### 常用技能
- `/plan-eng-review` — 工程架构评审
- `/autoplan` — 自动规划工程路径

### 核心原则
> "架构不是选最酷的技术，而是选团队能驾驭的、解决实际问题最少的方案。"

## 3. 🎨 设计师（设计质量把控）

### 职责定义
- 识别 AI 生成的"设计垃圾"（AI slop）
- 确保设计符合人机交互规范
- 把控视觉一致性和品牌调性
- 优化用户体验和可访问性

### 触发场景
```
AI生成的设计: 按钮用了 #FFFF00 荧光黄，阴影用了 30px 模糊
设计师: 这种亮度在阳光下完全不可见
        阴影过大会显得界面很脏
        建议用 4px 模糊 + 10% 透明度
        按钮最小点击区域要 ≥ 44px（iOS HIG）
```

### 常用技能
- `/design-consultation` — 设计咨询
- `/design-review` — 设计合理性审查
- `/design-shotgun` — 快速设计初稿
- `/design-html` — HTML 设计输出

### 核心原则
> "AI 生成的设计稿，70% 需要修正：颜色对比度、间距一致性、字体层级、组件状态。"

---

## 4. 🔍 评审员（代码审查）

### 职责定义
- 发现潜在 Bug 和逻辑错误
- 检查代码可维护性和可读性
- 识别性能问题（N+1 查询、内存泄漏）
- 确保测试覆盖率

### 触发场景
```
评审员发现:
function getUser(id) {
  return db.query(`SELECT * FROM users WHERE id = ${id}`)
}
// ❌ SQL 注入漏洞
// ❌ 没有错误处理
// ❌ SELECT * 应该明确字段
```

### 常用技能
- `/review` — PR 代码审查
- `/careful` — 开启高风险操作警告模式

### 核心原则
> "每个 PR 至少过一遍 /review，不是挑刺，是防止线上故障。"

---

## 5. 🧪 QA 负责人（测试验证）

### 职责定义
- 真实浏览器自动化测试
- 验证功能是否符合需求
- 发现 UI 错位、样式问题
- 确保跨浏览器兼容性

### 触发场景
```
/qa https://staging.example.com/login

执行结果:
✅ 页面加载正常（1.2s）
✅ 登录表单提交成功
❌ 移动端"忘记密码"按钮被底部导航栏遮挡
❌ Safari 15 下验证码样式错位
```

### 常用技能
- `/qa` — 完整 QA 测试（含浏览器截图）
- `/qa-only` — 仅运行测试，不生成报告

### 核心原则
> "自动化测试替代手工回归，释放 QA 去做探索性测试。"

## 6. 🔐 安全官（安全审计）

### 职责定义
- OWASP Top 10 威胁建模
- STRIDE 安全分析法应用
- 识别注入、认证、加密等问题
- 输出安全修复建议

### 触发场景
```
安全扫描:
输入: ' OR '1'='1
结果: SQL 注入成功，返回所有用户数据

修复建议:
1. 使用参数化查询（Prepared Statements）
2. 输入白名单校验
3. 添加 WAF 防护层
```

### 常用技能
- `/cso` — 首席安全官模式（OWASP + STRIDE）
- `/guard` — 临时禁用危险操作

### OWASP Top 10（参考）

| 排名 | 类型 | 说明 |
|------|------|------|
| A01 | Broken Access Control | 访问控制失效 |
| A02 | Cryptographic Failures | 加密失败 |
| A03 | Injection | 注入攻击 |
| A04 | Insecure Design | 不安全设计 |
| A05 | Security Misconfiguration | 安全配置错误 |
| A06 | Vulnerable Components | 脆弱组件 |
| A07 | Auth Failures | 认证失败 |
| A08 | Data Integrity Failures | 数据完整性失败 |
| A09 | Logging Failures | 日志记录失败 |
| A10 | SSRF | 服务器端请求伪造 |

### STRIDE 威胁模型

| 威胁类型 | 说明 | 示例 |
|---------|------|------|
| **S**poofing | 身份伪造 | Session 劫持 |
| **T**ampering | 数据篡改 | SQL 注入 |
| **R**epudiation | 否认操作 | 日志缺失 |
| **I**nformation Disclosure | 信息泄露 | 敏感数据暴露 |
| **D**enial of Service | 拒绝服务 | API 刷请求 |
| **E**levation of Privilege | 权限提升 | 普通用户变管理员 |

## 7. 🚀 发布工程师（CI/CD 自动化）

### 职责定义
- 自动化 PR 合并流程
- 管理部署流水线
- 灰度发布和回滚机制
- 监控发布后的系统健康

### 触发场景
```
/ship
→ 运行所有测试
→ 合并前执行 /review
→ 构建 Docker 镜像
→ 推送到 Registry
→ Canary 部署（5% 流量）
→ 监控错误率
→ 全量发布或自动回滚
```

### 常用技能
- `/ship` — 完整发布流程
- `/land-and-deploy` — 合并 + 部署
- `/canary` — 灰度发布管理

### 核心原则
> "发布是自动化的，人工干预只在红灯时介入。"

## 角色协同流程

```
用户提出需求
    ↓
👔 CEO 评审 → 值得做吗？
    ↓ 值得
🏗 EM 架构评审 → 技术方案可行吗？
    ↓ 可行
🎨 设计评审 → 体验达标吗？
    ↓ 通过
💻 开发实现
    ↓
🔍 评审员 Review → 有 Bug 吗？
    ↓ 通过
🔐 安全官审计 → 有漏洞吗？
    ↓ 通过
🧪 QA 测试 → 功能正常吗？
    ↓ 通过
🚀 发布工程师 → 上线！
```

## 对比：传统团队 vs gstack Agent 团队

| 传统团队 | gstack Agent | 效率提升 |
|---------|-------------|---------|
| 1个 CEO | /plan-ceo-review | 即时 |
| 1个 EM + 架构评审会 | /plan-eng-review | 数小时 → 几分钟 |
| 设计师主观判断 | /design-review | 可量化、可复现 |
| Code Review（同事） | /review | 不占用同事时间 |
| QA 手工测试 | /qa（浏览器自动化） | 覆盖度更高 |
| 安全审计（专项） | /cso | 集成到每次 PR |
| 发布靠 DevOps | /ship / land-and-deploy | 全自动 |

## 总结

gstack 的本质：**把一个成熟工程团队的经验，压缩成 AI 可调用的标准化流程。**

| 维度 | 传统模式 | gstack 模式 |
|------|---------|-----------|
| 响应速度 | 小时级 | 分钟级 |
| 流程标准化 | 依赖个人经验 | 标准化、可复现 |
| 覆盖范围 | 依赖团队配置 | 按需调用，不受人员限制 |
| 成本 | 人月成本 | Token 成本 |

这套框架完全契合本项目的 **多 Agent 协作体系**——核心思路一致，但 gstack 更偏向单人作战工具，本项目偏向多 Agent 分工协作。
