# 灵枢 Bot 多指令并发场景交互策略

> 版本：V1.0
> 日期：2026-04-01
> 状态：需求文档

---

## 一、问题背景

### 1.1 问题现象

```
时间线：
T0: 用户发送 命令A
     Bot回复：📥 已收到，开始分析...

T1: 用户发送 命令B  ← 此时A还在执行
     Bot应该怎么响应？

T2: 命令A执行中...
     Bot收到 命令C
     Bot应该怎么响应？

T3: 命令A完成
     命令B、C还在队列中
     Bot应该怎么执行？
```

### 1.2 用户痛点

| 用户心理 | 系统实际 |
|----------|----------|
| 每条消息都很重要 | 系统不知道优先级 |
| 希望立即响应 | 正在执行其他任务 |
| 发了多条就是急了 | 以为是重复发送 |
| 想取消之前的命令 | 没有取消机制 |
| 等待时焦虑 | 不知道要等多久 |

---

## 二、核心矛盾

```
用户期望：
1. 发送命令后立即得到确认（知道发成功了）
2. 执行期间可以发送新命令
3. 知道当前任务的状态和进度
4. 能够干预（取消、暂停、调整优先级）
5. 不想等待时可以插队

系统现状（openclaw）：
1. 无即时确认
2. 消息串行处理
3. 无状态反馈
4. 无干预机制
5. 无队列管理
```

---

## 三、设计目标

| 目标 | 说明 |
|------|------|
| **即时确认** | 收到命令后 <100ms 内回复 |
| **状态透明** | 用户随时知道任务进度 |
| **灵活干预** | 用户可以取消、暂停、调整优先级 |
| **智能调度** | 系统自动优化执行顺序 |
| **体验流畅** | 用户感觉不到阻塞 |

---

## 四、执行模型

### 4.1 基础执行模型

```
用户消息 → 即时ACK → 任务队列 → 异步执行 → 结果推送
     ↓
  <100ms   状态更新    多Worker     结果
```

### 4.2 任务状态机

```
PENDING → RUNNING → COMPLETED
    ↓          ↓
    ↓       FAILED
    ↓          ↓
    ↓       RETRY (可选)
    ↓
    CANCELLED (用户取消)
```

## 五、响应策略

### 5.1 消息分类与响应

| 消息类型 | Bot响应 | 延迟要求 |
|----------|---------|----------|
| **取消命令** | 「正在取消...」 | <500ms |
| **简单问候** | 「你好！」 | <1s |
| **紧急问题** | 「好的，马上处理」 | <1s |
| **状态查询** | 显示当前任务状态 | <1s |
| **正常任务** | 「📥 已收到，预计X分钟」 | <2s |

### 5.2 执行优先级规则

| 优先级 | 命令类型 | 处理方式 |
|--------|----------|----------|
| P0 | 取消/停止 | 立即中断当前任务 |
| P1 | 简单指令 (<1分钟) | 可打断长任务 |
| P2 | 复杂任务 (>1分钟) | 排队执行 |
| P3 | 低优先级 | 可延迟执行 |

### 5.3 任务执行中的响应策略

```
场景：正在执行命令A（预计3分钟）

用户发送命令B：
    ↓
判断B的类型
    ↓
┌─────────────────┬─────────────────┬─────────────────┐
│  B是取消/停止   │  B是简单指令    │  B是复杂任务    │
│      ↓          │      ↓          │      ↓         │
│  立即中断A      │  立即响应B      │  排队执行B     │
│  执行B          │  A继续执行      │  A继续执行     │
└─────────────────┴─────────────────┴─────────────────┘
```

## 六、多命令处理机制

### 6.1 命令合并

```
场景：用户连续发送多条相关命令

T0: 发送 命令A
T1: 发送 命令B  ← 与A相关
T2: 发送 命令C  ← 独立任务

Bot检测：命令A和B可能是同一任务的不同步骤
    ↓
Bot回复：
「📋 检测到相关命令，已自动合并
📋 命令C 已单独排队

当前状态：
🔄 任务1：代码分析+review (50%)
⏳ 任务2：架构设计（排队中）
」
```

### 6.2 重复命令过滤

```
场景：用户发送了相似的命令

T0: 用户发送：帮我review这段代码
T1: 用户又发送：帮我review这段代码 ← 重复

Bot检测：命令A和B完全相同
    ↓
Bot回复：
「📋 检测到重复命令，已自动忽略第2条

🔄 当前任务：代码分析 (进行中)

如需重新执行，请先「取消」当前任务。」
```

```

   • 短任务（<1分钟）将优先处理
   • 发送「状态」查看任务进度
   • 发送「取消」停止当前任务

```

```
│                                     │

│  💬 输入 "优先3" 可提升任务3优先级    │
└─────────────────────────────────────┘
```

### 7.3 任务执行进度推送

```
Bot主动汇报（不需要用户查询）：

T0: 开始执行
    → 「🚀 开始分析代码...」

T1: 30%进度
    → 「⚙️ 分析中... (30%)」

T2: 60%进度
    → 「⚙️ 风险检测中... (60%)」

T3: 完成
    → 「✅ 分析完成！」
       「📊 代码质量：85分」
       「🛡️ 安全风险：低」
       「⚡ 性能建议：3条」
```

### 7.4 短任务插队确认

```
场景：正在执行代码分析（预计3分钟）

用户发送：帮我写一封邮件（30秒就能完成）

Bot判断：
- 邮件是简单任务（<1分钟）
- 代码分析是复杂任务（>3分钟）

Bot回复：
「📋 检测到简短任务：写邮件（预计30秒）

选项：
[1] 立即执行（暂停当前任务）
[2] 继续执行当前任务，邮件排队列
[3] 合并到当前任务末尾

请回复数字选择，或忽略我继续执行当前任务。」
```

## 八、干预机制

### 8.1 干预命令

| 命令 | 效果 |
|------|------|
| 「取消」 | 停止当前任务 |
| 「取消2」 | 停止队列中第2个任务 |
| 「先回答这个」 | 打断，优先处理新任务 |
| 「等它完成」 | 继续当前，不被打断 |
| 「状态」 | 查看当前队列状态 |
| 「优先3」 | 提升任务3的优先级 |
| 「合并1和3」 | 合并任务1和3 |

### 8.2 干预确认流程

```
用户输入：「取消」

Bot回复：
「⚠️ 确定要取消当前任务吗？

任务：代码分析
进度：50%
已耗时：1分30秒

[确认取消] [继续执行」
```

## 九、智能队列管理

### 9.1 动态优先级调整

```
用户行为学习：
- 用户经常在执行长任务时发送短指令 → 短任务默认优先
- 用户经常取消当前任务 → 降低打断门槛
- 用户等待耐心低 → 提示预估时间

系统自动调整：
如果检测到用户等待焦虑（发送多次、表情反馈）
→ 自动加速或提供干预选项
```

### 9.2 预估时间提示

```
每次收到任务时，Bot告诉用户：

「📥 已收到：代码分析
⏱️ 预计完成时间：3分钟

💡 提示：
   • 执行期间您可以发送其他命令
   • 短任务（<1分钟）将优先处理
   • 发送「状态」查看任务进度
   • 发送「取消」停止当前任务
」
```

## 十、技术实现要点

### 10.1 即时ACK机制

```typescript
async function handleMessage(msg) {
  // 1. 立即ACK（<100ms）
  await sendAck(msg.chat_id, "📥 已收到消息");
  
  // 2. 判断任务类型
  const taskType = classifyTask(msg.content);
  
  // 3. 根据类型决定处理方式
  if (taskType === 'cancel') {
    await cancelCurrentTask();
    await sendResult(msg.chat_id, "✅ 已取消");
  } else if (taskType === 'simple') {
    await executeImmediately(msg);
  } else {
    await queueTask(msg);
    await sendQueueStatus(msg.chat_id);
  }
}
```

### 10.2 命令去重

```typescript
const recentCommands = new Map<string, number>();

function isDuplicate(userId, command) {
  const key = `${userId}:${hash(command)}`;
  const lastTime = recentCommands.get(key);
  
  if (lastTime && Date.now() - lastTime < 30000) {
    return true; // 30秒内相同命令视为重复
  }
  
  recentCommands.set(key, Date.now());
  return false;
}
```

### 10.3 任务队列结构

```typescript
interface Task {
  id: string;
  userId: string;
  type: 'simple' | 'complex' | 'background';
  priority: number;
  status: 'pending' | 'running' | 'completed' | 'failed' | 'cancelled';
  progress: number;
  createdAt: Date;
  estimatedTime?: number;
  result?: any;
}
```

## 十一、与openclaw的集成

### 11.1 当前openclaw的限制

| 问题 | openclaw现状 |
|------|--------------|
| 即时ACK | 无，收到后直接处理 |
| 任务队列 | 无，所有消息串行处理 |
| 进度反馈 | 无，只返回最终结果 |
| 去重机制 | 无，可能重复执行 |
| 干预机制 | 无，无法取消 |

### 11.2 灵枢的改进架构

```
┌─────────────────────────────────────────────────────────┐
│                    灵枢客户端层                         │
├─────────────────────────────────────────────────────────┤
│  用户消息 → 即时ACK → 任务队列 → 执行 → 结果推送         │
│      ↓                                                │
│  ┌─────────────────────────────────────────────────┐   │
│  │           任务状态管理                            │   │
│  │   状态：PENDING/RUNNING/COMPLETED/FAILED        │   │
│  │   进度：百分比 + 当前步骤                         │   │
│  │   去重：30秒内相同命令自动拦截                     │   │
│  └─────────────────────────────────────────────────┘   │
│      ↓                                                │
│  ┌─────────────────────────────────────────────────┐   │
│  │           openclaw 执行层                         │   │
│  │   Agent调度 + Skill执行 + 结果返回                 │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

## 十二、总结

### 12.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **即时确认** | 收到命令后 <100ms 内回复 |
| **状态透明** | 用户随时知道任务进度 |

### 12.2 交互策略总结

| 场景 | 推荐策略 |
|------|----------|
| 多条短命令 | 并行执行 |
| 长命令 + 短命令 | 短命令优先 |
| 重复命令 | 自动合并/忽略 |
| 用户想中断 | 立即中断 + 确认 |
| 长时间等待 | 定期进度推送 |
