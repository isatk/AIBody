## 二、灵犀引擎（Lingxi）技术设计

### 2.1 OpenClaw集成架构

#### 2.1.1 服务层与OpenClaw通信协议

```
┌─────────────────────────────────────────────────────────┐
│                  灵枢服务层 (Go)                        │
│   API网关 │ 任务队列 │ 即时ACK │ 会话管理              │
└─────────────────────┬─────────────────────────────────┘
                        │ gRPC (内部) / HTTP REST
                        ↓
┌─────────────────────────────────────────────────────────┐
│                  灵犀引擎 (OpenClaw)                    │
│   Gateway │ Agent Runtime │ Memory │ Skills             │
└─────────────────────────────────────────────────────────┘
```

| 通信方式 | 协议 | 用途 | 延迟要求 |
|----------|------|------|----------|
| **gRPC** | Protobuf | 内部高效通信 | <10ms |
| **HTTP REST** | JSON | 简单请求/响应 | <50ms |
| **WebSocket** | JSON | 实时推送 | <100ms |

#### 2.1.2 Agent会话管理

```
用户 ↔️ 客户端
    ↓ WebSocket
灵枢服务层 (Go)
    ↓ gRPC
OpenClaw Gateway
    ↓
Agent Runtime (智能体调度)
    ↓
Skill Executor / Agent Executor
    ↓
结果返回

会话流程：
1. 用户发起请求 → 服务层接收
2. 服务层鉴权 → 路由到对应Agent
3. Agent加载上下文 → 执行任务
4. 实时日志推送 → 进度反馈
5. 执行完成 → 结果汇总 → 推送用户
```

#### 2.1.3 上下文记忆系统

| 记忆类型 | 存储 | 生命周期 | 容量 |
|----------|------|-----------|------|
| **短期记忆** | Redis | 会话内 | 4K-128K tokens |
| **长期记忆** | SQLite | 持久化 | 无限制 |
| **跨会话记忆** | 向量数据库 | 用户生命周期 | 无限制 |
| **知识记忆** | Vector Search | 永久 | 按需检索 |

```
记忆检索流程：
用户请求 → 提取关键词 → 向量相似度搜索 → 召回相关记忆 → 注入上下文 → Agent处理
```

路由流程：

┌─────────────────────────────────────────────────────────┐

                          ↓

#### 2.2.2 并行执行机制

```
场景：用户需要同时完成代码review和文档撰写

串行执行：
A(review) ──────→ B(文档) ──────→ 结果
总计：10分钟

并行执行：
A(review) ──────→ 结果
B(文档)   ──────→ 结果
总计：5分钟

并行条件：
- 任务间无依赖
- 系统资源充足
- 用户未指定顺序
```

#### 2.2.3 安全边界三层防护

| 层级 | 机制 | 说明 |
|------|------|------|
| **/careful** | 谨慎模式 | 高风险操作需二次确认 |
| **/freeze** | 冻结模式 | 禁止执行高危操作 |
| **/guard** | 监护模式 | 所有操作记录审计日志 |

```
安全边界示例：

/careful 模式：
- 删除操作需要输入「确认」才能执行
- 高危API调用前提示警告

/freeze 模式：
- 禁止文件删除
- 禁止网络请求到外部
- 禁止系统命令执行

/guard 模式：
- 记录所有操作日志
- 操作可追溯
- 异常行为告警
```

## 四、灵犀客户端通信协议

### 4.1 实时通信

#### 4.1.1 WebSocket连接管理

```
连接建立流程：
1. 客户端发起WebSocket连接
2. 服务端验证JWT Token
3. 建立长连接
4. 推送连接成功消息

心跳机制：
- 客户端每30秒发送心跳
- 服务端60秒无响应则断开
- 断线自动重连（最多3次）
```

#### 4.1.2 消息确认（ACK）与顺序保证

| 机制 | 说明 |
|------|------|
| **消息ACK** | 每条消息需要确认送达 |
| **序列号** | 消息有序号，乱序需重排 |
| **重发机制** | ACK超时未收到则重发 |

```
ACK流程：
1. 客户端发送消息（带seq_id）
2. 服务端收到后返回ACK（seq_id）
3. 若超时未收到ACK，客户端重发
```

### 4.2 多端同步

#### 4.2.1 消息同步协议

| 场景 | 同步策略 |
|------|----------|
| **新消息** | 实时推送 + 存储 |
| **历史消息** | 按需拉取（分页） |
| **多设备** | 服务端为唯一数据源 |

#### 4.2.2 已读/未读同步

```
已读同步流程：
1. 用户A阅读消息 → 发送已读ACK
2. 服务端更新消息状态
3. 同步给用户A的其他设备
4. 通知消息发送者

未读计数：
- 每个会话独立计数
- 推送通知携带未读数
```

#### 6.2.3 智能体调用链路追踪

```
追踪结构：
TraceID → SpanID → ParentSpanID
   ↓
用户请求ID → 服务层处理 → 引擎调用 → 智能体执行 → 技能调用

关键字段：
- trace_id: 全链路唯一ID
- span_id: 当前操作ID
- parent_id: 上级操作ID
- service_name: 服务名称
- operation_name: 操作名称
- start_time / end_time: 时间
- tags: 附加信息
```
