## 二、为什么不是纯Go或纯Node.js

### 2.1 纯Go重写的代价

| 项目 | 成本 |
|------|------|
| 工程量 | 1-2年+ |
| 插件生态 | 需要全部重写 |
| AI SDK | Go生态远不如Node.js |
| 社区支持 | 失去openclaw社区 |

### 2.2 纯Node.js的局限

| 项目 | 问题 |
|------|------|
| API性能 | 弱于Go |
| 并发处理 | 不如Go |
| 部署 | 依赖Node.js环境 |

### 2.3 混合架构优势

| 优势 | 说明 |
|------|------|
| 性能最优 | Go处理高并发API |
| 能力完整 | 复用openclaw全部生态 |
| 开发效率 | 快速上线 |
| 维护成本 | 社区活跃 |

## 三、各层技术栈

### 3.1 客户端层

| 平台 | 技术 |
|------|------|
| iOS | Swift + SwiftUI |
| Android | Kotlin + Jetpack Compose |
| macOS | Swift + SwiftUI |
| Windows/Linux | Tauri 2.0 |
| Web | React + TypeScript |

### 3.2 服务层（Go）

| 组件 | 技术 |
|------|------|
| API框架 | Go + Gin |
| WebSocket | gorilla/websocket |
| 任务队列 | Redis + 自研 |
| ORM | GORM |
| 缓存 | Redis Cluster |
| 文件存储 | MinIO |
| 通信 | gRPC |

### 3.3 执行层（Node.js + openclaw）

| 组件 | 技术 |
|------|------|
| 运行时 | Node.js >= 22.12 |
| Web框架 | Hono |
| AI SDK | OpenAI / Anthropic |
| Agent | pi-agent-core |
| 向量存储 | sqlite-vec |
| 协议 | MCP SDK |

## 四、openclaw在架构中的角色

### 4.1 openclaw提供的能力

```
openclaw执行层
├── Gateway（AI请求入口）
├── Agent Runtime（多Agent调度）
├── Memory System（向量记忆）
├── Plugin SDK（扩展接口）
├── Skills（gstack 23技能）
└── Agents（agency 193智能体）
```

### 4.2 灵枢服务层与openclaw的通信

```
Go服务层                    openclaw执行层
    │                              │
    │  ──── HTTP/gRPC ────→       │
    │        发送任务               │
    │                              │
    │  ←─── 实时日志 ────         │
    │        进度推送               │
    │                              │
    │  ←─── 执行结果 ────         │

    │                              │
```
